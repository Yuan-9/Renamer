# 照片视频重命名软件需求分析

## 1. 项目背景

本项目是一个面向 Windows 桌面的照片、视频批量重命名工具。主要使用场景是整理相机、手机、运动相机、无人机等设备拍摄的照片和视频，将文件名统一改为基于拍摄时间的格式，方便归档、排序、查找和备份。

应用计划使用 JavaScript + React + Electron 开发 Windows 桌面界面。Electron 提供本地文件访问、系统对话框、子进程调用和桌面打包能力，适合本项目快速实现文件扫描、元数据读取和批量重命名。

## 2. 核心目标

1. 批量读取用户选择文件夹中的照片和视频文件。
2. 根据文件的拍摄时间生成新文件名。
3. 只改变文件名，不修改文件内容，不改写照片或视频的原始元数据。
4. 在重命名前提供清晰预览，让用户确认旧文件名、新文件名、时间来源和冲突状态。
5. 对重名冲突进行稳定、可预测的序号处理。
6. 支持 Windows 用户常见的拖拽、文件夹选择、撤销记录和失败提示。
7. 支持两种重命名模式：原地重命名、重命名后移动到指定目录。

## 3. 命名格式

用户期望的命名格式：

```text
年份_日期_时分秒_毫秒_冲突序号
```

默认落地格式建议：

```text
YYYY_MMDD_HHmmss_SSS_NN.ext
```

示例：

```text
2026_0517_193025_128_00.jpg
2026_0517_193025_128_01.jpg
2026_0517_193026_004_00.mp4
```

字段说明：

| 字段 | 示例 | 说明 |
| --- | --- | --- |
| 年份 | `2026` | 四位年份 |
| 日期 | `0517` | 月日，固定 4 位 |
| 时分秒 | `193025` | 24 小时制，固定 6 位 |
| 毫秒 | `128` | 固定 3 位，不足补零 |
| 冲突序号 | `00` | 同一时间戳下的顺序号，固定至少 2 位，从 `00` 开始 |
| 扩展名 | `.jpg` | 保留原格式，扩展名大小写可由用户设置 |

需要后续确认的细节：

1. 日期字段是否固定为 `MMDD`，还是希望使用 `YYYY_MMDD` 之外的完整日期格式，例如 `YYYY_MM_DD`。

固定规则：

1. 冲突序号始终写入文件名，即使没有冲突也显示 `_00`。

## 4. 命名模板

应用必须支持命名模板。默认模板为：

```text
{yyyy}_{MMdd}_{HHmmss}_{SSS}_{index}
```

默认生成结果：

```text
2026_0517_193025_128_00.jpg
```

模板变量建议：

| 变量 | 含义 | 示例 |
| --- | --- | --- |
| `{yyyy}` | 四位年份 | `2026` |
| `{MM}` | 两位月份 | `05` |
| `{dd}` | 两位日期 | `17` |
| `{MMdd}` | 月日组合 | `0517` |
| `{HH}` | 两位小时，24 小时制 | `19` |
| `{mm}` | 两位分钟 | `30` |
| `{ss}` | 两位秒 | `25` |
| `{HHmmss}` | 时分秒组合 | `193025` |
| `{SSS}` | 三位毫秒 | `128` |
| `{index}` | 冲突序号，从 `00` 开始 | `00` |
| `{original}` | 原文件名，不含扩展名 | `IMG_0001` |

模板规则：

1. 文件扩展名不写入模板，由应用根据原文件扩展名自动追加。
2. 模板需要实时预览，用户修改模板后立即重新生成新名称。
3. 模板中不能包含 Windows 文件名非法字符：`< > : " / \ | ? *`。
4. 模板为空或解析失败时，禁止执行重命名，并在设置区域提示原因。

## 5. 文件范围

首期建议支持以下常见格式。

照片：

```text
.jpg, .jpeg, .png, .heic, .heif, .tif, .tiff, .webp, .dng, .raw, .cr2, .cr3, .nef, .arw, .orf, .rw2
```

视频：

```text
.mp4, .mov, .m4v, .avi, .mkv, .wmv, .mts, .m2ts, .3gp
```

注意：

1. 不同格式的拍摄时间字段差异较大，尤其是 RAW、HEIC、MOV、MP4。
2. 应用需要允许用户过滤照片、视频或两者都处理。
3. 不支持的格式默认跳过，并在预览中显示原因。

## 6. 时间来源规则

拍摄时间应按可信度从高到低读取：

1. 照片 EXIF `DateTimeOriginal`。
2. 照片 EXIF `CreateDate` / `ModifyDate`。
3. 照片 EXIF 亚秒字段，例如 `SubSecTimeOriginal`，用于补充毫秒。
4. 视频容器元数据中的创建时间，例如 QuickTime / MP4 `creation_time`。
5. 文件系统创建时间。
6. 文件系统修改时间。

预览列表中必须显示实际使用的时间来源，例如：

```text
EXIF DateTimeOriginal
EXIF CreateDate
Video creation_time
File created time
File modified time
```

如果无法读取任何可靠时间，文件应进入“待处理异常”状态，不自动生成最终文件名，除非用户选择允许使用文件修改时间兜底。

## 7. 毫秒处理

毫秒字段优先从元数据的亚秒字段读取。

处理规则：

1. 元数据有精确毫秒：直接使用。
2. 元数据只有一位或两位亚秒：右侧补零到 3 位。
3. 元数据没有毫秒但文件系统时间有毫秒：可使用文件系统毫秒，并标记时间来源。
4. 完全没有毫秒：使用 `000`。

示例：

| 原始亚秒 | 结果 |
| --- | --- |
| `1` | `100` |
| `12` | `120` |
| `128` | `128` |
| 无 | `000` |

## 8. 冲突序号规则

冲突包括两类：

1. 多个源文件生成同一个目标文件名。
2. 目标目录中已经存在同名文件。

建议规则：

1. 同一目标目录内按拍摄时间升序、原文件名升序排序。
2. 第一份文件使用 `_00`。
3. 同时间戳的后续文件依次使用 `_01`、`_02`。
4. 如果目标目录已有同名文件，则继续递增序号，直到找到可用名称。
5. 冲突序号宽度默认 2 位，超过 99 后自动扩展为 3 位。
6. 冲突序号始终显示，不因没有冲突而省略。

## 9. 重命名行为

应用只执行文件系统层面的重命名或移动操作：

```text
old-path/file-name.ext -> old-path/new-name.ext
old-path/file-name.ext -> output-path/new-name.ext
```

必须保证：

1. 不打开文件进行写入。
2. 不改写 EXIF、XMP、IPTC、视频容器元数据。
3. 不压缩、不转码、不改变文件内容。
4. 不改变文件扩展名所代表的格式。
5. 默认使用原地重命名模式。

重命名模式：

1. 原地重命名：文件保留在原目录，只修改文件名。
2. 重命名后移动到指定目录：先按规则生成新文件名，再移动到用户指定的输出目录。
3. 移动到指定目录时默认不保留原目录结构，后续可作为高级选项支持保留目录结构。
4. 目标目录不存在时应提示用户创建或重新选择，不应静默失败。

必须支持的设置：

1. 扩展名大小写：保持原样、统一小写、统一大写。
2. 输出方式：原地重命名、重命名后移动到指定目录。
3. 重命名日志：每次执行都生成日志，便于手动恢复和后续撤销。
4. 命名模板：允许用户编辑模板，并保存最近一次使用的模板。

## 10. 桌面应用功能

首期功能：

1. 选择一个文件夹作为输入目录。
2. 快速读取文件夹中的文件名并扫描支持的照片、视频文件。
3. 支持递归扫描子文件夹。
4. 根据默认模板或用户模板生成修改后的名称。
5. 以列表形式左右对比展示：左侧原文件名，右侧新文件名。
6. 列表中同时展示类型、时间、时间来源、状态和提示信息。
7. 不显示缩略图。
8. 支持按状态过滤：可重命名、冲突、失败、跳过。
9. 支持重命名模式切换：原地重命名、重命名后移动到指定目录。
10. 执行批量重命名。
11. 显示执行结果：成功、失败、跳过。
12. 生成并导出操作日志。

建议增强功能：

1. 撤销上一次重命名。
2. 保存用户偏好设置。
3. 支持预设模板切换。
4. 支持深色模式。
5. 支持中英文界面。

## 11. 用户界面初步结构

主界面建议分为以下区域：

1. 顶部工具栏：选择文件夹、重新扫描、清空列表、设置。
2. 设置区：命名模板、扩展名大小写、递归扫描、是否使用兜底时间。
3. 重命名模式区：原地重命名、移动到指定目录；仅选择移动模式时显示输出目录选择。
4. 中央预览表格：左侧展示原文件名，右侧展示新文件名，并显示时间来源和状态。
5. 底部操作栏：文件统计、执行重命名按钮、导出日志按钮。

界面明确不显示缩略图，主视图优先保证文件名读取速度和列表对比效率。

## 13. 数据模型

单个文件在应用内部可抽象为：

```ts
type RenameItem = {
  id: string;
  originalPath: string;
  directory: string;
  originalName: string;
  extension: string;
  mediaType: "photo" | "video" | "unknown";
  capturedAt: string | null;
  millisecond: number;
  timeSource: string | null;
  proposedName: string | null;
  proposedPath: string | null;
  renameMode: "in-place" | "move-to-directory";
  outputDirectory: string | null;
  conflictIndex: number | null;
  status: "ready" | "conflict" | "warning" | "error" | "skipped" | "renamed";
  message?: string;
};
```

应用设置可抽象为：

```ts
type RenameSettings = {
  template: string;
  extensionCase: "preserve" | "lower" | "upper";
  renameMode: "in-place" | "move-to-directory";
  outputDirectory: string | null;
  recursive: boolean;
  useModifiedTimeFallback: boolean;
};
```

## 14. 技术需求

桌面壳：

1. 最终技术选型确定为 Electron + React。
2. Electron 主进程负责文件系统操作、系统文件夹选择、ExifTool 调用和批量重命名执行。
3. React 渲染进程负责设置表单、左右对比列表、状态过滤和执行结果展示。
4. 主进程和渲染进程之间通过安全 IPC 通信传递扫描结果、预览结果和执行状态。

元数据读取：

1. 可优先考虑调用 ExifTool，它对照片和视频格式支持最完整。
2. 应用需要处理 ExifTool 不存在、执行失败或读取超时的情况。
3. 后续可将 ExifTool 随应用打包，避免要求用户手动安装。

文件操作：

1. 使用 Node.js `fs.rename` 或等价 API。
2. 重命名前再次检查目标路径是否存在。
3. 对大小写仅变化的重命名需要兼容 Windows 文件系统行为。
4. 失败时不继续覆盖文件，记录错误并处理下一个文件。

## 15. 安全与可靠性

1. 执行前必须预览。
2. 默认禁止覆盖已有文件。
3. 原地重命名模式默认保留原目录结构。
4. 默认跳过无法确定时间的文件。
5. 每次执行生成日志，记录原路径、新路径、时间、状态和错误。
6. 撤销功能依赖日志反向重命名，执行前仍需检查冲突。
7. 大批量文件处理需要进度条和可取消机制。
8. 移动到指定目录时，执行前必须检查目标目录写入权限和目标文件冲突。

## 16. 非功能需求

性能：

1. 需要支持至少数千个文件的批量预览。
2. 元数据读取应异步并发，但限制并发数量，避免系统卡顿。
3. UI 表格应支持虚拟滚动，避免大量文件时渲染过慢。

兼容性：

1. Windows 10 及以上。
2. 处理中文路径、空格路径、长路径。
3. 保留原始文件格式；目录是否保留取决于重命名模式。

可维护性：

1. 重命名规则、时间提取、文件扫描、桌面 UI 分层实现。
2. 关键规则需要单元测试覆盖。
3. 文件操作需要集成测试或沙盒目录测试。

## 17. 首期验收标准

1. 用户可以选择一个包含照片和视频的文件夹。
2. 应用可以快速读取文件夹内文件名，并扫描支持的照片、视频格式。
3. 应用可以读取拍摄时间并生成预览名称。
4. 预览以左右对比列表展示原文件名和新文件名。
5. 文件名默认格式符合 `YYYY_MMDD_HHmmss_SSS_NN.ext`，其中冲突序号从 `00` 开始。
6. 用户可以修改命名模板，并实时看到新名称变化。
7. 同名冲突可以自动生成递增序号。
8. 用户可以选择扩展名大小写策略。
9. 用户可以选择原地重命名或重命名后移动到指定目录。
10. 点击执行后只改变文件名或文件路径，不改写文件内容或元数据。
11. 执行完成后显示成功和失败数量。
12. 每次执行生成可读的操作日志。
